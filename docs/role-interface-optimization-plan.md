## 第一章：概述

### 1.1 项目背景

射洪建设支撑小程序（SNYD N/W 支撑）是一套面向通信运营商网络建设全流程的工单管理系统，覆盖需求发起、设计查勘、施工管理、监理验收、开通确认的完整链路。

当前系统已有 21 个页面、10 个角色定义（8 主要角色 + 2 兼容角色），但存在以下问题：

- 所有角色共用同一套底部导航栏（首页 / 工单 / 消息 / 我的），未根据角色职能差异化
- 首页快捷入口虽已按角色分支，但核心内容区（统计卡片、待办列表）缺乏角色定制
- 区县经理、部门经理在需求详情页中无任何操作按钮，角色感知弱
- 部分角色缺少专属的任务列表入口（如网格经理的待审核入口分散）

### 1.2 优化目标

本次优化不改动现有 21 个页面的代码逻辑，仅通过以下方式提升各角色的使用体验：

1. **底部导航栏按角色差异化**：Tab2（工单/任务/验收/管理）根据当前角色动态切换标签名和目标页面
2. **首页看板按角色定制**：每个角色看到不同的统计维度、快捷入口、待办列表
3. **新增区县经理和部门经理角色界面**：补全这两个角色在原 PRD 中缺失的界面描述
4. **角色-页面权限矩阵规范化**：明确每个角色可访问的页面及可执行的操作

### 1.3 设计原则

| 原则 | 说明 |
|------|------|
| 4-Tab 统一结构 | 保持微信小程序底部导航 4 Tab 惯例，Tab1/3/4 固定为首页/消息/我的，Tab2 按角色变化 |
| 角色职能优先 | 界面设计服务于角色在工单流转中的职责，而非覆盖所有功能 |
| 操作路径最短 | 核心高频操作在首页 Tab2 首屏可见，减少点击层级 |
| 数据范围隔离 | 每个角色只能看到自己权限范围内的工单数据（由后端 API 过滤） |

## 第三章：底部导航栏配置

### 3.1 总体配置原则

所有角色保持 4-Tab 底部导航结构，Tab1（首页）、Tab3（消息）、Tab4（我的）标签名固定不变，Tab2（第二个 Tab 位）按角色动态切换：

| Tab 序号 | 固定标签名 | 说明 |
|---------|-----------|------|
| Tab1 | 首页 | 角色专属看板 |
| Tab2 | **按角色变化** | 核心业务入口（工单/任务/验收/管理） |
| Tab3 | 消息 | 系统通知和工单通知 |
| Tab4 | 我的 | 个人信息和设置 |

### 3.2 各角色 Tab2 配置详情

| 角色 | Tab2 标签名 | Tab2 图标 | Tab2 目标页面 | Tab2 职能说明 |
|------|-----------|----------|-------------|--------------|
| 一线人员 | 工单 | demand | demand-list | 查看自己提交的工单 |
| 设计 | 任务 | task | design-task | 处理分配给自己的设计任务 |
| 施工 | 任务 | task | construction-task | 处理分配给自己的施工任务 |
| 监理 | 验收 | verify | supervisor-task | 处理待验收的工单 |
| 网格经理 | 工单 | demand | demand-list | 查看网格内的工单 |
| 网络支撑经理 | 工单 | demand | demand-list | 查看区域内的工单 |
| 区县经理 | 工单 | demand | demand-list | 查看所有工单 |
| 部门经理 | 工单 | demand | demand-list | 查看所有工单 |
| 四级经理 | 工单 | demand | demand-list | 查看所有工单 |
| 管理员 | 管理 | admin | admin | 进入管理后台 |

### 3.3 技术实现方案

基于现有的 `custom-tab-bar` 组件实现动态 Tab2 切换。

**关键实现思路**：在 `custom-tab-bar/index.js` 的 `setData` 方法中，读取 `app.globalData.currentRole`，根据角色动态替换 `list[1]`（Tab2）的 `text` 和 `pagePath` 字段。

现有 `custom-tab-bar/index.js` 使用内联 SVG data URI 实现图标，无需额外资源文件。只需在 `switch` 语句中为不同角色返回不同的 tabList 数组即可。

### 4.5 关键交互流程

**流程 1：提交新需求**
```
首页 → [新建需求] → 需求录入页（填表单+地图选点+拍照）→ 提交 → 跳转需求详情 → 查看进度
```

**流程 2：被驳回后重新提交**
```
消息通知 → [点击查看] → 需求详情（显示驳回原因）→ [重新提交] → 需求录入页（编辑模式，锁定区域字段）→ 修改内容 → 重新提交
```

## 第十六章：技术实现建议

### 16.1 底部导航栏动态切换

**实现位置**：`miniprogram/custom-tab-bar/index.js`

**实现思路**：在组件的 `setData` 中读取 `app.globalData.currentRole`，根据角色动态生成 tabList。

```javascript
// 伪代码示例
const tabConfig = {
  FRONTLINE: [{ text: '工单', icon: 'demand', path: 'pages/demand-list/demand-list' }],
  DESIGN: [{ text: '任务', icon: 'task', path: 'pages/design-task/design-task' }],
  CONSTRUCTION: [{ text: '任务', icon: 'task', path: 'pages/construction-task/construction-task' }],
  SUPERVISOR: [{ text: '验收', icon: 'verify', path: 'pages/supervisor-task/supervisor-task' }],
  ADMIN: [{ text: '管理', icon: 'admin', path: 'pages/admin/admin' }],
  // ... 其他角色默认用 '工单'
};

// 读取当前角色，替换 tabList[1]（Tab2）
```

现有 `custom-tab-bar/index.js` 已使用内联 SVG data URI 作为图标，无需新增图标文件。

### 16.2 首页看板按角色渲染

**实现位置**：`miniprogram/pages/home/home.js`

现有 `buildShortcuts()` 方法（第 49-96 行）已按角色返回不同的快捷入口配置。可在此基础上扩展：

1. **新增 `buildStats()` 方法**：返回不同角色的统计数据维度和数值
2. **新增 `buildPendingList()` 方法**：返回不同角色的待办列表内容（工单卡片）
3. **新增 `buildExtraSection()` 方法**：返回不同角色的辅助信息（如服务中心进度表）

### 16.3 页面权限守卫

对于试图通过 URL 直接访问未授权页面的用户，建议在每个页面的 `onLoad` 中加入角色检查。不具有访问权限的角色尝试访问时，调用 `wx.redirectTo` 跳转回首页。

```javascript
// 伪代码示例
const roleAccessMap = {
  'pages/design-task/design-task': ['DESIGN'],
  'pages/construction-task/construction-task': ['CONSTRUCTION'],
  'pages/supervisor-task/supervisor-task': ['SUPERVISOR'],
  'pages/admin/admin': ['ADMIN'],
  // ...
};

onLoad(options) {
  const currentRole = getApp().globalData.currentRole;
  const pagePath = 'pages/design-task/design-task'; // 当前页面
  if (roleAccessMap[pagePath] && !roleAccessMap[pagePath].includes(currentRole)) {
    wx.redirectTo({ url: '/pages/home/home' });
    return;
  }
}
```
