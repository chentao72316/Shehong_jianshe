1. 项目背景与目标
1.1 背景
地市公司一线人员在日常工作中需要发起网络建设需求，现有流程依赖线下沟通、邮件流转，存在以下问题：
- 流程不透明，需求发起后无法追踪进度
- 信息分散，设计图纸、施工照片散落在聊天记录中
- 协作效率低，跨单位沟通成本高
- 工期不可控，超时无人跟进
1.2 目标
- 实现建设需求从录入 → 设计 → 施工 → 反馈的全流程线上化
- 通过飞书多维表格实现数据统一存储与管理
- 通过飞书群消息实现关键节点的自动推送与提醒
- 建立工期标准与超时预警机制，确保建设周期可控
- 采用手机号认证，确保只有授权人员可访问系统
- 提升流程透明度与跨单位协作效率

3. 工期标准与超时机制
3.1 工期标准
环节标准工期说明设计查勘2天从需求指派给设计单位开始计算建设交付5天从施工指派时间开始计算（设计完成后系统自动指派施工）整体建设周期≤7天从需求提交到施工完成的总时长
3.1.1 工期计算规则
- 计算单位：自然日（包含周末和节假日）
- 起算时间：以系统记录的指派时间为起点，精确到小时
- 特殊说明：如需按工作日计算，可在工期配置中将标准工期适当延长
- 节假日豁免：管理员可在系统设置中配置节假日豁免期间，豁免期内暂停超时计算
3.2 时间节点定义
需求提交 ──┬──→ 设计指派 ──→ 设计完成 ──→ 施工指派 ──→ 施工完成
           │                │                    │               │
           │                │                    │               │
           └────── 2天 ─────┘                    └──── 5天 ─────┘
           │
           └────────────── 整体 ≤7天 ─────────────────┘

3.3 超时判断规则
超时类型判断条件触发动作设计超时设计指派后 > 2天 且 状态≠“待施工”设计超时预警施工超时施工开始后 > 5天 且 状态≠”已开通”施工超时预警整体超时需求提交后 > 7天 且 状态≠“已开通”整体超时预警 + 升级督办
3.4 超时状态
状态值说明设计中（超时）设计环节超过2天未完成施工中（超时）施工环节超过5天未完成整体超时总周期超过7天未完工

6.2 驳回流程
当设计单位判断”300米内有资源，无需建设”时：
- 填写驳回原因（现有现场资源名称、备注）
- 需求状态变更为”已驳回”
- 触发飞书群驳回通知，@提交人 + @网格经理 + @网络支撑经理
6.3 驳回后重新提交流程
当需求被驳回后，一线人员可修改并重新提交：
- 一线人员在需求详情页可查看驳回原因（包括现有资源名称、驳回备注）
- 点击”重新提交”进入需求修改页
- 可修改字段：需求类型、已预约客户数、拟建DP箱个数、拟建位置（地图）、拟建位置详细描述、现场照片、紧急程度、备注
- 不可修改字段：受理区域、服务网格、提交人电话（保持原值）
- 重新提交后：
  - 需求状态从”已驳回”变更为”设计中”
  - 驳回次数 +1
  - 重新记录设计指派时间（工期从重新提交时重新计算）
  - 系统自动按原区域重新指派设计单位（或沿用原设计人员）
  - 触发飞书消息通知设计单位重新查勘

9. 超时预警与督办模块
9.1 超时检测机制
定时任务配置：
- 执行频率：每小时执行一次（普通/紧急需求）
- 特急需求：每30分钟执行一次超时检测
- 即时检测：需求状态变更时触发一次即时超时检测
- 检测范围：所有状态为”设计中”或”施工中”的需求
9.2 超时判断逻辑
// 超时判断伪代码
function checkTimeout(demand) {
  const now = new Date();
  
  // 设计超时判断
  if (demand.status === '设计中') {
    const designHours = (now - demand.designAssignTime) / (1000 * 3600);
    if (designHours > 48) { // 超过2天
      return { type: 'design', level: 'warning' };
    }
  }
  
  // 施工超时判断
  if (demand.status === '施工中') {
    const constructionHours = (now - demand.constructionAssignTime) / (1000 * 3600);
    if (constructionHours > 120) { // 超过5天
      return { type: 'construction', level: 'warning' };
    }
  }
  
  // 整体超时判断
  const totalHours = (now - demand.submitTime) / (1000 * 3600);
  if (totalHours > 168 && demand.status !== '已开通') { // 超过7天
    return { type: 'total', level: 'urgent' };
  }
  
  return null;
}

9.3 超时预警级别
级别触发条件通知对象处理要求普通预警设计 > 1.5天（36小时）设计单位提醒加快进度超时警告设计 > 2天（48小时）设计单位 + 网格经理 + 网络支撑经理督办处理紧急督办施工 > 4天（96小时）施工单位 + 网格经理 + 网络支撑经理 + 监理限时处理升级督办整体 > 7天（168小时）全体相关人员 + 管理员升级处理
9.4 超时提醒节点
时间节点触发动作消息类型设计第1.5天（36小时）首次预警设计即将超时提醒设计第2天（48小时）超时警告设计超时督办通知设计第2.5天（60小时）二次督办设计严重超时通知施工第4天（96小时）紧急督办施工紧急督办通知施工第5天（120小时）超时警告施工超时督办通知整体第7天（168小时）升级督办整体超时升级通知
9.5 督办任务管理
督办任务列表（网格经理/网络支撑经理/监理可见）：
字段说明工单编号关联需求超时类型设计超时/施工超时/整体超时超时时长超过标准工期的时长当前责任人设计单位/施工单位督办状态待处理/处理中/已处理操作发送督办消息、查看详情、强制干预
9.6 流程干预操作定义
"流程干预"是指网格经理、网络支撑经理或管理员对异常流程进行人工介入的操作，具体包括：
- 重新指派人员：将当前设计/施工任务指派给其他人员（从人员配置表中选择同区域的其他可用人员）
- 强制变更状态：在特殊情况下（如需求作废、客户取消等），将需求状态强制变更为"已开通"或"已驳回"
- 添加督办备注：对当前流程添加督办说明，记录干预原因和处理措施
操作权限：
- 网格经理：仅限重新指派和添加督办备注
- 网络支撑经理：重新指派、强制变更状态、添加督办备注
- 管理员：所有干预操作
所有流程干预操作均记入督办记录表，并触发飞书群消息通知相关人员。
9.7 进度计算规则
需求列表和详情页中的进度百分比按以下规则计算：
- 待指派：0%
- 设计中：20%
- 待施工：40%
- 施工中：60%
- 已开通：100%
- 已驳回：不显示进度

// 手机号认证登录流程
async function loginByPhone() {
  // 1. 获取微信授权手机号
  const phoneNumber = await wx.getPhoneNumber();
  
  // 2. 查询飞书人员配置表
  const userRecords = await feishuTable.query({
    table: "人员配置表",
    filter: `手机号 = "${phoneNumber}"`
  });
  
  // 3. 判断是否有权限
  if (userRecords.length === 0) {
    showError("您的手机号未授权，请联系管理员");
    return;
  }
  
  // 4. 获取用户角色
  const roles = extractRoles(userRecords);
  
  // 5. 处理多角色情况
  if (roles.length > 1) {
    const selectedRole = await showRoleSelectPage(roles);
    return selectedRole;
  }
  
  // 6. 返回用户信息和角色
  return {
    phoneNumber,
    roles: roles[0],
    userInfo: userRecords[0]
  };
}
